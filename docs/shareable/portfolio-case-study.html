Why ARR is hard

ARR looks easy until billing intervals, discounts, churn, reactivation, renewals, and exclusions all collide.

The project makes those rules explicit instead of hiding them in dashboards.

Certified monthly ARR

Expected monthly totals are fixture-backed and validated in the repo.

Proof stack

1. Contract

Who owns the metric, what it means, and what it is not.

2. Logic

Eligibility, effective dates, annualization, and exclusions.

3. Evidence

Expected answers are hand-calculated before implementation.

4. Controls

Singular tests, unit tests, model contracts, and semantic validation.

Architecture at a glance

Seeds → Staging → Intermediate ARR logic → Certified mart → Semantic layer → BI / AI

The point is to certify once and reuse many times.

Snowflake proof

Validation is shown by two things:

  • the semantic view DDL exists in the repo;
  • the validation queries prove the object and its outputs in Snowflake.

The strongest evidence is a query screenshot or exported result set showing the semantic view, exposed metrics, and month-end totals matching the fixtures.
